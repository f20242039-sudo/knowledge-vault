# Knowledge Vault — Project Constitution

## Mission
Build an AI-powered knowledge management system that lets users upload documents and automatically discover skills, topics, and relevance through semantic AI analysis — replacing manual tagging with intelligent organization.

---

## Core Principles

### 1. AI-First, Not Search-First
The system must use AI to *understand* documents, not just index keywords. Every document gets a structured tag profile generated by Claude — not by the user.

### 2. User Uploads, AI Organizes
Users should never have to manually assign categories, difficulty levels, or skill tags. The AI does this automatically on upload.

### 3. Retrieval Over Summary
The system surfaces *which documents* are relevant to a query, not a summary of those documents. The raw document remains the source of truth.

### 4. Transparency in Tagging
Every AI-generated tag must include a confidence/coverage level (e.g. "covers 80% of this skill") so users understand why a document was returned.

### 5. Modular Pipeline
The document processing pipeline (extract → tag → embed → store) must be fully decoupled from the UI. Each stage is independently testable.

---

## Coding Standards

### Language & Runtime
- **Python 3.11+** throughout
- **Streamlit** for UI — no custom frontend frameworks
- **Type hints** on all functions
- **Docstrings** on all modules and public functions

### File & Folder Conventions
- Snake_case for all Python files and variables
- One responsibility per module (extractors, taggers, embedders, db clients are separate)
- No business logic inside Streamlit `app.py` — it only calls service modules

### AI / Prompt Engineering
- All Claude prompts live in `ai/prompts/` as `.txt` or `.py` constants — never inline in business logic
- Prompts must request **structured JSON output** with a defined schema
- Always validate Claude's JSON response against a Pydantic model before storing

### Database
- All Supabase queries go through `db/supabase_client.py` — no raw SQL elsewhere
- Schema migrations are tracked in `db/migrations/`
- Vector column uses `pgvector` extension — embeddings are 384-dimensional (sentence-transformers default)

### Error Handling
- File parsing errors must be caught and surfaced to the user in the UI — never silently fail
- Claude API errors must trigger a retry (max 2) before surfacing to the user
- Failed uploads are logged but do not crash the session

### Testing
- Every pipeline step (extract, tag, embed, store) has a unit test in `tests/`
- Test files use `pytest`
- Mock the Claude API and Supabase in unit tests — no live calls in CI

---

## Quality Rules

- No hardcoded API keys — all secrets via `.env` and `python-dotenv`
- No file over 300 lines — split into submodules if needed
- Every PR must include a test for any new pipeline step
- UI must handle empty states gracefully (no documents yet, no results, failed upload)
- All tag schemas must be versioned — changing the tag structure requires a migration

---

## Definition of Done
A feature is done when:
1. It works end-to-end in the Streamlit UI
2. It has unit tests for all pipeline logic
3. Error states are handled and visible to the user
4. The relevant spec/plan/tasks docs are updated
